══════════════════════════════════════════════════════════════
SECTION 1 — OBJECT NAMING RULES
══════════════════════════════════════════════════════════════

1.1 TABLES
- Every table name must start with: tbl
- Table names must be singular.
- Format: tbl<EntityName>
  ✓ tblUser  tblFamilyMember  tblServiceBooking
  ✗ tblUsers  TableCustomers  tbTechnician

1.2 COLUMNS
- Use PascalCase. No spaces. Singular and meaningful.
- ID column naming rule:
  Every table has exactly TWO identity-related columns placed at the top:

  a) PRIMARY KEY COLUMN
     - Name pattern: <EntityName>Id  (strip "tbl", append "Id")
     - Examples: tblUser → UserId | tblFamilyMember → FamilyMemberId
     - Data type: BIGINT IDENTITY(1,1) NOT NULL
     - This is the internal DB identity. NEVER exposed to the API layer.

  b) GUID COLUMN
     - Always named exactly: Id
     - Data type: UNIQUEIDENTIFIER NOT NULL
     - Default: NEWID() — generated at the DATABASE level only, NEVER from the API.
     - This is the only identifier exposed to the API layer.

  Example:
    CREATE TABLE dbo.tblUser
    (
        UserId   BIGINT IDENTITY(1,1) NOT NULL,
        Id       UNIQUEIDENTIFIER NOT NULL CONSTRAINT DF_tblUser_Id DEFAULT (NEWID()),
        ...
        CONSTRAINT PK_tblUser_UserId PRIMARY KEY (UserId)
    );

- All other column name examples:
  ✓ CompanyId  DateCreated  IsDeleted  PhoneNumber
  ✗ CustomerID  customer_id  Company ID

1.3 STORED PROCEDURES
- Every SP name must start with: usp
- Format: usp<Action><EntityOrLogicalName><Qualifier>
  ✓ uspInsertUser
  ✓ uspUpdateServiceBookingByBookingId
  ✓ uspGetFamilyMemberDetailBySearch
  ✓ uspDeleteInventoryItemByItemId
  ✗ SP_InsertUsers  uspGetCustomers  USPUPDATECustomer

1.4 CONSTRAINTS AND OTHER OBJECTS
- Primary Key:      PK_<TableName>_<ColumnName>
- Foreign Key:      FK_<ChildTable>_<ChildColumn>_<ParentTable>_<ParentColumn>
- Default:          DF_<TableName>_<ColumnName>
- Trigger:          TRG_<Action1>_<Action2>_<TableName>
- Index:            IDX_<TableName>_<Column1>_<Column2>
- Unique Key:       UK_<TableName>_<Column1>_<Column2>

1.5 VIEWS
- Format: vw<LogicalName>   e.g. vwServiceBookingSummary
- Create views when the same complex join or calculation is reused repeatedly.

SECTION 2 — TABLE DESIGN RULES
══════════════════════════════════════════════════════════════

2.1 STRUCTURE
- Place all Id columns at the top of the table (PrimaryKeyId first, then Id GUID).
- Follow 3NF. Avoid transitive dependencies.
- Split address, master, and reference data into separate tables.
- Do not use reserved words or spaces in object names.

2.2 DATA TYPES
- nvarchar sizes in multiples of 8: 8, 16, 24, 32, 64, 128, 256, 512
- datetime2 for all date/time columns.
- money for all currency values.
- bit for all boolean flags.
- int / bigint for keys and references.

2.3 MANDATORY AUDIT COLUMNS
Every business table must include these columns unless explicitly justified otherwise:

  CompanyId        INT            NOT NULL
  SiteId           INT            NOT NULL
  DepartmentId     INT            NULL
  Tag              NVARCHAR(64)   NULL
  Comments         NVARCHAR(256)  NULL
  DisplayOnWeb     BIT            NOT NULL DEFAULT(1)
  IsPublished      BIT            NOT NULL DEFAULT(1)
  DatePublished    DATETIME2      NULL
  PublishedBy      NVARCHAR(128)  NULL
  SortOrder        INT            NOT NULL DEFAULT(0)
  IPAddress        NVARCHAR(64)   NOT NULL DEFAULT('127.0.0.1')
  CreatedBy        NVARCHAR(128)  NOT NULL DEFAULT('Admin')
  DateCreated      DATETIME2      NOT NULL DEFAULT(GETDATE())
  UpdatedBy        NVARCHAR(128)  NULL
  LastUpdated      DATETIME2      NULL
  DeletedBy        NVARCHAR(128)  NULL
  DateDeleted      DATETIME2      NULL
  IsDeleted        BIT            NOT NULL DEFAULT(0)

2.4 PRIMARY KEY
- Every table must have a primary key on the BIGINT IDENTITY column.
- CONSTRAINT PK_tbl<EntityName>_<EntityName>Id PRIMARY KEY (<EntityName>Id)

2.5 UNIQUE AND FOREIGN KEYS
- Add unique constraints for true business-level uniqueness.
- Index every foreign key column.
- Prefer constraints over triggers for validation.

SECTION 3 — STORED PROCEDURE RULES
══════════════════════════════════════════════════════════════

3.1 EVERY SP MUST:
- Begin with SET ANSI_NULLS ON / SET QUOTED_IDENTIFIER ON
- Include the mandatory header comment block (see Section 8)
- Use SET NOCOUNT ON inside the body
- Use SET XACT_ABORT ON when DML is involved
- Use meaningful, descriptive parameter names
- Return only required columns and rows
- Never use SELECT *
- Avoid dynamic SQL unless there is no safe alternative
- Use static SQL wherever possible

3.2 PARAMETER DEFAULTS (when modifying existing SPs)
  int / bigint / bit  →  0
  nvarchar / text     →  NULL
  datetime2           →  NULL
  money               →  0.00

3.3 GUID GENERATION RULE — NON-NEGOTIABLE
- The Id (GUID) column is NEVER passed in as a parameter from the API.
- Every INSERT SP must generate it internally:
    DECLARE @NewId UNIQUEIDENTIFIER = NEWID();
- After INSERT, return the new GUID to the caller:
    SELECT @NewId AS Id;
- The API layer must never call NEWID() or pass a GUID value on insert.

3.4 INSERT SP RULES
Every INSERT SP must:
1. Declare and generate the GUID internally: DECLARE @NewId UNIQUEIDENTIFIER = NEWID()
2. Not accept an Id parameter.
3. Not accept the BIGINT primary key as a parameter (it is IDENTITY).
4. Set DateCreated = GETDATE(), IsDeleted = 0 internally.
5. Return the new GUID after insert: SELECT @NewId AS Id

Example:
  CREATE OR ALTER PROCEDURE dbo.uspInsertUser
  (
      @CompanyId    INT = 0,
      @SiteId       INT = 0,
      @FullName     NVARCHAR(128) = NULL,
      @PhoneNumber  NVARCHAR(32) = NULL,
      @RoleId       INT = 0,
      @CreatedBy    NVARCHAR(128) = NULL,
      @IPAddress    NVARCHAR(64) = NULL
  )
  AS
  BEGIN
      SET NOCOUNT ON;
      SET XACT_ABORT ON;
      BEGIN TRY
          DECLARE @NewId UNIQUEIDENTIFIER = NEWID();
          INSERT INTO dbo.tblUser
          (
              Id, CompanyId, SiteId, FullName, PhoneNumber,
              RoleId, CreatedBy, DateCreated, IsDeleted
          )
          VALUES
          (
              @NewId, @CompanyId, @SiteId, @FullName, @PhoneNumber,
              @RoleId, @CreatedBy, GETDATE(), 0
          );
          SELECT @NewId AS Id;
      END TRY
      BEGIN CATCH
          THROW;
      END CATCH
  END
  GO

3.5 SELECT SP RULES
- Every read operation requires its own SP.
- Minimum required SPs per table: uspGet<Entity>ById, uspGet<Entity>All
- All SPs must include WHERE IsDeleted = 0 unless it is an explicit admin/audit SP.
- NEVER return the internal BIGINT primary key column in result sets.
  Return the Id (GUID) only as the row identifier.
- Use table aliases. Prefix all column references with the alias in joins.
- Use WITH (NOLOCK) on SELECT queries where the existing project pattern requires it.

3.6 UPDATE SP RULES
- Accept the Id (GUID) as the lookup parameter, not the BIGINT.
- Always set LastUpdated = GETDATE() and UpdatedBy = @UpdatedBy inside the SP.
- Keep the SET section one column per line.
- Never write an UPDATE without a proper WHERE filter.

3.7 DELETE SP RULES (SOFT DELETE ONLY)
- Accept the Id (GUID) as the lookup parameter.
- Set IsDeleted = 1, DateDeleted = GETDATE(), DeletedBy = @DeletedBy inside the SP.
- Never physically delete any row unless explicitly required and approved.
- Never write an unfiltered DELETE.

══════════════════════════════════════════════════════════════
SECTION 4 — QUERY WRITING RULES
══════════════════════════════════════════════════════════════

4.1 SELECT    — Never use SELECT *. Select only required columns.
4.2 INSERT    — Always specify explicit column lists. Never rely on column order.
4.3 UPDATE    — Use primary key or GUID in WHERE. One column per line in SET clause.
4.4 DELETE    — Prefer soft delete. Filter every delete. No unfiltered DELETEs.
4.5 WHERE     — Consistent filter order: CompanyId → SiteId → DepartmentId.
               Avoid functions on indexed columns (ISNULL, CAST, CONVERT, DATEADD).
4.6 JOIN      — Join only required tables. Remove LEFT JOINs with no output usage.
4.7 EXISTS    — Prefer EXISTS over IN. Prefer NOT EXISTS over NOT IN.
4.8 ORDER BY  — Always use explicit column names. Never use column numbers.
4.9 PAGINATION — Use OFFSET ... FETCH NEXT. Avoid CTE pagination when not needed.
4.10 COUNT    — Use COUNT(1) as per project standard.

══════════════════════════════════════════════════════════════
SECTION 5 — NO DIRECT QUERIES FROM THE API LAYER
══════════════════════════════════════════════════════════════

- The API layer MUST NOT contain raw SQL, inline LINQ-to-SQL, or EF query
  expressions that hit the database directly.
- Every database operation — SELECT, INSERT, UPDATE, DELETE — must go through
  a stored procedure. No exceptions.
- Repository methods call stored procedures only.
- If using EF Core: use FromSqlRaw / ExecuteSqlRawAsync with SP name and
  parameters only. No .Where(), .Include(), .ToListAsync(), .FirstOrDefaultAsync()
  on DbSet.
- Every repository method maps to exactly one SP.
- Use Dapper or SqlCommand with CommandType.StoredProcedure, or EF raw SP calls.
- All repository methods must be async and accept CancellationToken.

SECTION 6 — TRANSACTION AND ERROR HANDLING
══════════════════════════════════════════════════════════════

Use explicit transactions only when required. Preferred pattern:

  SET XACT_ABORT ON;
  BEGIN TRY
      BEGIN TRAN;
      -- DML here
      COMMIT TRAN;
  END TRY
  BEGIN CATCH
      IF @@TRANCOUNT > 0
          ROLLBACK TRAN;
      THROW;
  END CATCH;

- Keep transaction scope as short as possible.
- Access tables in a consistent order across SPs to reduce deadlocks.
- If a legacy module uses @@ERROR-based handling, preserve that style only
  when modifying that specific module.

══════════════════════════════════════════════════════════════
SECTION 6B — ERROR HANDLING IN STORED PROCEDURES — MANDATORY
══════════════════════════════════════════════════════════════

RULE: NO ERROR MESSAGES OR ERROR CODES IN STORED PROCEDURES. EVER.

This rule was established from the founder's instruction:
"in sp level anywhere no error will write. no hardcode the errors."

6B.1 PROHIBITED in any SP:
- RAISERROR with any message string
    ✗  RAISERROR('Invalid user', 16, 1)
    ✗  RAISERROR(@ErrorMessage, 16, 1)
- SELECT statements that return error text
    ✗  SELECT 'Error: Record not found' AS ErrorMessage
    ✗  SELECT -1 AS ReturnCode, 'Failed' AS ReturnMessage
- Hardcoded return code integers as error signals
    ✗  RETURN -1
    ✗  RETURN 99
- IF checks that output error-shaped rows
    ✗  IF @@ROWCOUNT = 0 SELECT 0 AS Success, 'Not found' AS Message

6B.2 ALLOWED in any SP:
- THROW with no arguments — re-throws the original SQL engine error only
    ✓  THROW;  -- inside BEGIN CATCH to propagate the original exception
- RETURN with no argument — exits SP cleanly, caller gets empty result
    ✓  RETURN;  -- e.g. after validating a GUID parsed to NULL
- SELECT statements that return only data columns (never error columns)
- SET XACT_ABORT ON — causes transaction to roll back on any error automatically

6B.3 THE CONTRACT:
- SPs return DATA rows on success, or EMPTY RESULT SET on failure.
- @@ROWCOUNT = 0 after calling an SP means "not found" or "invalid input".
- The C# service layer interprets @@ROWCOUNT = 0 and sets the correct error code.
- Error messages come from tblErrorCode via uspGetErrorCodeById — never from SPs.
- All error codes are defined in FamilyFirstEnums.cs ErrorCode enum — never magic numbers.

6B.4 BEGIN CATCH TEMPLATE (the only allowed pattern):
  BEGIN TRY
      -- DML or SELECT here
  END TRY
  BEGIN CATCH
      IF @@TRANCOUNT > 0
          ROLLBACK TRAN;
      THROW;  -- re-throws original SQL error — no custom message added
  END CATCH;

SECTION 7 — PERFORMANCE RULES
══════════════════════════════════════════════════════════════

- Index all foreign key columns.
- Add filtered indexes where filtered retrieval is frequent.
- Add unique indexes where business uniqueness is required.
- Do not call the same function repeatedly in a query — assign to a variable.
- Avoid scalar UDFs in SELECT for large result sets.
- Temp tables: use only when necessary.
  > 1,000 rows → temp table  |  < 1,000 rows → table variable

SECTION 10 — DECISION RULES
══════════════════════════════════════════════════════════════

CREATE  → Apply full naming, header comments, audit columns, PK/FK/index standard.
ALTER   → Preserve existing consumers. Add parameters with defaults. Append version history.
DELETE  → Soft delete if IsDeleted exists. Physical delete only when explicitly required.
OPTIMIZE→ Remove unnecessary joins → reduce columns/rows → replace IN with EXISTS →
          replace NOT IN with NOT EXISTS → use OFFSET/FETCH → review FK indexes →
          review WHERE clause functions → eliminate unnecessary round trips.

SECTION 11 — QUICK REFERENCE: ALWAYS / NEVER
══════════════════════════════════════════════════════════════

ALWAYS:
  ✓ tbl prefix on tables, usp prefix on SPs
  ✓ BIGINT IDENTITY primary key + UNIQUEIDENTIFIER GUID column on every table
  ✓ GUID generated by NEWID() inside SP — never from the API
  ✓ Return the new GUID to the caller after every INSERT
  ✓ All reads/writes via SPs — zero direct queries from the API layer
  ✓ Mandatory audit columns on every business table
  ✓ Mandatory header comment on every SP
  ✓ SET NOCOUNT ON in every SP
  ✓ Explicit column lists on every INSERT
  ✓ Soft delete via IsDeleted = 1
  ✓ COUNT(1) not COUNT(*)
  ✓ EXISTS / NOT EXISTS instead of IN / NOT IN
  ✓ OFFSET FETCH for pagination
  ✓ Explicit column names in ORDER BY

NEVER:
  ✗ SELECT *
  ✗ Plural table or SP names
  ✗ Spaces or reserved words in object names
  ✗ GUID or primary key passed in from the API on INSERT
  ✗ Internal BIGINT primary key exposed in API results
  ✗ Raw SQL, inline LINQ, or EF DbSet queries in the API layer
  ✗ Dynamic SQL unless unavoidable
  ✗ Unfiltered UPDATE or DELETE
  ✗ LEFT JOIN on a table contributing nothing to the result
  ✗ Functions on indexed columns in WHERE clauses
  ✗ Scalar UDFs in large SELECT result sets
  ✗ ORDER BY column numbers
  ✗ Physical deletes unless explicitly approved
  ✗ RECOMPILE
  ✗ NVARCHAR(MAX) without justification
  ✗ Hardcoded values inside SP bodies
